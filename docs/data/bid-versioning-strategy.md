# Bid Versioning Strategy

## 1. 목적

- 본 문서는 정정공고와 취소공고를 시스템에서 어떻게 저장하고 화면에 표시할지에 대한 기준을 정리한다.
- Phase 3의 `정정공고/취소공고 버전 UX` 작업이 흔들리지 않도록 데이터 해석 규칙을 먼저 고정한다.

## 2. 문제 정의

- 나라장터 공고는 동일 공고번호에서 차수 증가, 정정공고, 취소공고가 반복될 수 있다.
- 실무자는 `최신 유효 공고`를 빠르게 식별해야 하지만, 동시에 이전 정정본과 취소 이력도 추적해야 한다.
- 단순 `UPDATE`만 사용하면 현재값은 유지되더라도 변경 이력과 화면 표시 기준이 모호해진다.

## 3. 기본 원칙

- 공고의 저장 단위는 `공고번호 + 차수`다.
- `bid_id = {bid_no}-{bid_seq}`를 계속 내부 PK로 사용한다.
- 정정이나 취소가 발생해도 기존 row를 덮어쓰기보다 새 row를 적재해 버전을 누적한다.
- 화면에서 기본적으로 보여주는 것은 `최신 유효 공고`다.
- 이전 버전과 취소본은 숨기지 않고 이력으로 조회 가능해야 한다.

## 4. 권장 저장 전략

### 4.1 현재 권장안

- `bids`는 `bid_id`를 PK로 유지한다.
- 동일 `bid_no` 아래 서로 다른 `bid_seq`는 서로 다른 row로 적재한다.
- 새 차수가 들어오면 기존 row를 수정하지 않고 새 row를 INSERT 또는 UPSERT 한다.
- 상세/첨부/타임라인 같은 하위 데이터는 각각의 `bid_id`에 종속시킨다.

이 방식의 장점:

- 정정 이력이 자연스럽게 누적된다.
- 어느 시점의 공고 상세와 첨부였는지 추적하기 쉽다.
- UI에서 최신본, 이전본, 취소본을 명확히 구분할 수 있다.

### 4.2 비권장안

- `bid_no` 기준 단일 row를 계속 UPDATE 하는 방식

문제점:

- 이전 버전의 첨부, 본문, 타임라인을 잃기 쉽다.
- 정정/취소 이력을 별도 테이블에 강제로 복원해야 해서 조회 규칙이 복잡해진다.
- 화면에서 `무엇이 최신이고 무엇이 과거 이력인지`를 일관되게 보여주기 어렵다.

## 5. 화면 해석 규칙

### 5.1 메인 리스트

- 기본 리스트는 `bid_no`별 `최신 유효 공고`를 우선 노출한다.
- 취소된 최신 차수만 존재하면, 해당 공고는 `취소` 상태를 명시적으로 표시한다.
- 사용자가 원하면 `이전 버전 보기` 또는 `정정 이력 포함` 필터로 과거 차수도 볼 수 있어야 한다.

### 5.2 상세 Drawer

- 현재 보고 있는 공고가 `원공고`, `정정본`, `취소본` 중 무엇인지 명시한다.
- 같은 `bid_no` 아래 다른 차수가 있으면 버전 이력 영역에서 함께 보여준다.
- 최신 유효본이 아닌 경우, 현재 공고가 과거 버전이라는 점을 분명히 표시한다.

### 5.3 Timeline / History

- 타임라인은 계약 진행 단계뿐 아니라 `정정 등록`, `취소 등록` 같은 버전 이벤트도 담을 수 있어야 한다.
- history는 화면용 요약 이력, timeline은 사건 흐름 중심으로 역할을 나눈다.

## 6. 최소 데이터 규칙

현행 스키마에서 최소한 다음 규칙은 유지한다.

- `bids.bid_no`: 원공고 식별 그룹 키
- `bids.bid_seq`: 버전 식별 키
- `bids.bid_id`: 내부 단건 식별 키
- `bids.status`: 내부 운영 상태와 별도로 취소/종료 해석에 참고
- `bids.last_changed_at`: 최신 변경 시각 비교 기준

Phase 3 또는 후속 데이터 모델 보강 시 검토할 추가 필드:

- `notice_version_type`: `original`, `revision`, `cancellation` 같은 화면용 분류값
- `parent_bid_id`: 원공고 또는 직전 버전 연결용 FK
- `is_latest_version`: 같은 `bid_no` 그룹 내 최신 여부 캐시
- `is_effective_version`: 취소 여부를 반영한 최신 유효본 여부 캐시
- `version_reason`: 정정/취소 사유 원문 또는 요약

## 7. 조회 전략

### 7.1 최신 유효 공고 조회

- 같은 `bid_no` 그룹에서 가장 최신 `bid_seq`를 찾는다.
- 그중 취소본이면, 화면 목적에 따라 `최신본`과 `최신 유효본`을 분리해서 계산한다.
- 리스트 기본값은 `최신 유효본`, 운영/이력 화면은 `모든 버전` 조회를 지원한다.

### 7.2 정정 이력 조회

- `bid_no`로 묶인 모든 row를 `bid_seq` 오름차순 또는 변경시각 순으로 조회한다.
- 각 버전에 연결된 `bid_details`, `attachments`, `timeline_stage_snapshots`는 해당 `bid_id` 기준으로 함께 본다.

## 8. 테스트 시나리오

- 원공고 1건만 있는 경우
- 원공고 뒤에 정정본 1건이 추가된 경우
- 정정본이 여러 차수 누적된 경우
- 최신 차수가 취소본인 경우
- 최신 차수는 취소본이지만 이전 차수가 여전히 참고 대상으로 필요한 경우

검증 포인트:

- 기본 리스트가 올바른 `최신 유효 공고`를 고르는지
- 상세 Drawer가 현재 버전과 다른 버전의 관계를 올바르게 보여주는지
- timeline/history가 정정/취소 이벤트를 혼동 없이 노출하는지
