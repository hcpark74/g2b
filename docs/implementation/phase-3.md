# Phase 3 Implementation Plan

`PLAN.md`의 Phase 3를 실제 구현 단위로 세분화한 실행 문서다.

## 1. 목표

- Phase 2에서 확보한 실제 수집 데이터를 실무형 검토 화면으로 연결한다.
- 메인 리스트, 필터, 상세 Drawer, 결과 화면을 업무 흐름 중심으로 정교화한다.
- 관심 공고, 내부 상태, 타임라인, 수동 액션을 운영 UI에서 일관되게 다룰 수 있게 한다.
- 나라장터의 시각적 친숙함은 유지하되, 실제 UX는 현대적인 데이터 그리드와 명확한 상태 표현 중심으로 개선한다.

## 2. 범위

포함:

- 메인 리스트 검색/필터/정렬 UX 고도화
- 상세 Drawer 정보 구조 재정리
- 관심 공고/내부 상태 변경 액션 보강
- 정정공고/취소공고의 버전 관계와 최신본 표시 규칙 정리
- timeline, history, 결과 화면 시각화 개선
- 사전 탐색/사후 분석 화면 초안
- Excel 내보내기와 운영자용 보조 액션 초안
- 크롤링/재수집 실패 이력과 health 상태를 운영 UI에 노출

제외:

- 외부 API 신규 확장 자체
- 배치 스케줄러/Zero Trust/배포 자동화
- PostgreSQL 운영 전환
- 권한 체계 분리와 멀티테넌시

## 4. 작업 단위

### Track A. 메인 리스트와 필터 UX 정교화

목표: 공고 검색과 선별을 실무 흐름에 맞게 빠르게 수행할 수 있게 한다.

결정:

- 메인 리스트의 기본 목적은 `빠른 후보 선별`로 둔다. 한 화면에서 검색, 필터, 정렬, 버전 상태, 관심 여부를 먼저 판독하고 상세 검토는 Drawer로 넘긴다.
- 기본 정렬은 `업데이트 최신순`으로 두고, 보조 정렬은 `마감일 빠른순`, `마감일 늦은순`, `공고일 최신순`, `기관명`, `입찰명` 정도로 제한한다.
- 페이지 크기 기본값은 `25`, 선택 옵션은 `25 / 50 / 100`으로 둔다. 리스트 상단에는 `현재 조건 결과 n건`을 항상 노출한다.
- 우선 컬럼은 `공고명`, `기관`, `상태`, `버전 상태`, `마감일`, `관심 여부`, `내부 상태`, `최종 갱신 시각` 순으로 두고, `공고명`과 `기관`은 좌측 고정 우선 컬럼으로 본다.
- 데이터 그리드는 Phase 3에서 전면 교체보다 `현재 HTMX 테이블 계약 유지 + 점진 강화`를 기본안으로 둔다.
- query param 계약에는 최소 `q`, `status`, `agency`, `date_from`, `date_to`, `favorite`, `sort`, `page`, `page_size`, `include_versions`를 포함한다.
- `include_versions` 기본값은 `false`로 두고, 기본 리스트는 `최신 유효본` 중심으로 유지한다.
- 로딩 표현은 리스트 영역 skeleton 또는 로딩 행을 우선하고, empty state는 `데이터 없음`과 `필터 결과 없음`을 구분한다.

체크리스트:

- [ ] 검색어, 상태, 기관, 기간, 관심 여부 필터 조합 재정리
- [ ] 결과 건수, 정렬, 페이지 크기, empty state 표현 보강
- [ ] 고정 컬럼/가로 스크롤/행 밀도 정책 확정
- [ ] AG Grid, Tabulator 등 데이터 그리드 도입 여부 검토
- [ ] HTMX partial 경계와 URL query 규칙 정리

완료 기준:

- 사용자가 메인 리스트에서 원하는 공고를 빠르게 좁혀 상세 검토로 진입할 수 있다.
- 나라장터와 유사한 정보 구조를 유지하면서도 정렬, 다중 필터, 컬럼 고정 같은 핵심 상호작용이 현대적으로 동작한다.

### Track B. 상세 Drawer 정보 구조 재정리

목표: Phase 2에서 수집한 상세 정보를 검토 우선순위에 맞게 재배치한다.

결정:

- 상세 Drawer의 역할은 `단건 검토의 중심 화면`으로 고정한다. 사용자가 메인 리스트를 떠나지 않고도 공고 판단에 필요한 핵심 맥락을 읽을 수 있어야 한다.
- 섹션 순서는 `overview -> qualification -> attachments -> timeline -> history`로 둔다.
- `overview`는 공고명, 기관, 공고번호/차수, 마감일, 계약 방식, 예산 또는 추정금액, 내부 상태, 관심 여부, 버전 상태를 우선 배치하는 요약 영역으로 사용한다.
- Drawer 헤더와 `overview` 상단에는 최소 `원공고/정정본/취소본`, `최신 유효본 여부`, 내부 상태, 관심 여부를 상태 배지로 명시한다.
- `qualification`은 참가 자격, 지역 제한, 업종/면허, 공동수급, 실적 제한처럼 입찰 가능 여부 판단에 직접 필요한 정보부터 보여준다.
- `attachments`는 첨부 개수, 주요 파일명, 파일 유형 또는 출처 메타데이터를 먼저 보여주고, 세부 목록은 그 아래에서 확인하게 한다.
- `timeline`은 계약 단계와 함께 `정정 등록`, `취소 등록`, `최신 유효본 변경` 같은 버전 이벤트를 시간순으로 보여주는 사건 흐름 영역으로 둔다.
- `history`는 버전별 핵심 차이, 계약 결과 요약, 최근 변경 요약을 카드나 목록으로 재정리하는 화면용 요약 이력으로 둔다.
- 크롤링 본문은 `본문 요약 우선, 원문은 보조` 원칙을 따른다. `crawl_data.text_summary`가 있으면 이를 우선 노출하고, 장문 원문은 접힘 영역으로 둔다.
- 계약과정통합공개 결과는 `overview` 안의 강조 카드 또는 상단 보조 패널로 노출해 낙찰업체, 낙찰금액, 계약일자, 계약명을 먼저 판독하게 한다.
- placeholder는 구조 유지용으로만 제한하고, 실제 데이터가 없을 때는 `미수집`, `정보 없음`, `아직 반영 전` 같은 의미 기반 문구를 사용한다.
- 현재 공고가 최신 유효본이 아니면 `과거 버전` 또는 `취소본` 안내를 상단에 강하게 배치하고, 같은 `bid_no`의 버전 흐름은 `차수 목록 + 현재 선택 + 최신 유효본 표시`로 함께 보여준다.

체크리스트:

- [ ] overview/qualification/attachments/timeline/history 섹션 우선순위 재정의
- [ ] 크롤링 본문, 계약 진행, 첨부 메타데이터 표시 규칙 정리
- [ ] placeholder와 실제 데이터 표시 조건 단순화
- [ ] 중요 값 누락 시 fallback 문구/강조 방식 정리
- [ ] 정정공고/취소공고 여부, 원공고 관계, 최신본 여부 표시 규칙 정리

완료 기준:

- 상세 Drawer만으로 공고 요약, 자격, 첨부, 진행 상태를 판독할 수 있다.
- 사용자가 현재 보고 있는 공고가 원본인지 정정본인지 취소본인지 즉시 구분할 수 있다.

### Track C. 업무 액션 연결

목표: 실무자가 공고 상태를 관리하고 필요한 재수집을 직접 수행할 수 있게 한다.

결정:

- 메인 리스트에서는 `관심 토글`과 `내부 상태 빠른 변경`만 허용하고, 현재 필터/정렬/페이지 상태를 유지한 채 HTMX partial로 목록만 갱신한다.
- 상세 Drawer 상단에는 `업무 액션` 블록을 두고 `관심 등록/해제`, `상태 변경`, `상세 재실행`, `계약과정 재실행`, `크롤링 재실행`, `운영 이력` 버튼을 제공한다.
- 액션 피드백은 별도 페이지 이동보다 현재 Drawer 내부 alert를 우선한다. 성공/실패 여부에 따라 색상과 메시지를 구분하고, 후속 추적 링크를 함께 보여준다.
- 수동 재실행은 모두 운영 액션으로 간주하고 `bid_detail_enrichment`, `contract_process_sync`, `bid_page_crawl` 작업 로그를 남긴다.
- 액션 직후에는 현재 화면에서 요약 결과만 보여주고, 상세 이력과 실패 원인은 `/operations?job_type=...`에서 확인하게 한다.
- sample backend와 sqlmodel backend 모두에서 동일한 액션 흐름을 유지해 UI 검증과 실데이터 검증의 차이를 최소화한다.
- 크롤링 health는 `정상`, `주의`, `실패` 3단계로 단순화한다. 최근 성공 시각과 최근 실패 시각이 있으면 이를 기준으로 상태를 판정하고, 화면 문구도 같은 기준을 사용한다.
- 크롤링 실패 이력은 개별 예외 전문을 그대로 노출하기보다 최근 실패 시각, 실패 단계, 요약 메시지, 최근 재시도 여부를 우선 보여준다. 상세 원인은 `/operations` 로그에서 확인하게 한다.
- 상세 Drawer에서는 `크롤링 재실행` 버튼 근처에 최근 크롤링 상태 요약을 두고, `/operations`에서는 `bid_page_crawl` 작업 이력을 기준으로 실패 반복 여부를 판독하게 한다.
- 리스트에서는 크롤링 health를 독립 컬럼으로 늘리기보다 상세 Drawer와 운영 화면에서 우선 노출한다. 메인 리스트는 선별 중심이라는 원칙을 유지한다.

체크리스트:

- [ ] 관심 공고 토글 UX 보강
- [ ] 내부 상태값 변경 액션 정리
- [ ] 수동 상세 보강/계약과정통합공개/크롤링 재실행 액션 초안
- [ ] 액션 성공/실패 피드백과 `/operations` 연결 규칙 정리
- [ ] 크롤링 실패 이력, 최근 재시도 횟수, health 상태 노출 규칙 정리

완료 기준:

- 리스트와 상세 화면에서 핵심 운영 액션을 끊김 없이 실행할 수 있다.
- 재수집 액션 이후 운영자가 실패 원인과 후속 상태를 화면에서 추적할 수 있다.

### Track D. 타임라인/결과 시각화 고도화

목표: 계약 진행과 결과 데이터를 화면에서 빠르게 판독할 수 있게 한다.

결정:

- Drawer timeline은 `상태 배지 + 사건 카드 + 진행 바` 구조를 기본으로 둔다. 계약 단계와 버전 이벤트를 같은 시간축에서 읽게 하되, 카드 단위로 사건 의미를 먼저 보이게 한다.
- inline timeline과 Drawer timeline 모두 `계약 진행과 버전 이벤트를 시간순으로 보여준다`는 설명을 유지해 화면 역할을 분명히 한다.
- history는 timeline의 반복이 아니라 `핵심 변경 요약` 영역으로 둔다. 항목명, 변경 전, 변경 후, 보조 문맥을 비교 중심으로 보여준다.
- `/results`는 계약 결과 목록 중심 화면으로 유지한다. 공고번호/업무구분, 공고명/기관, 공고 상태 배지, 버전 배지, 낙찰금액, 계약금액, 계약명을 한 화면에서 판독하게 한다.
- 결과 화면의 상태 문구와 버전 문구는 메인 리스트/Drawer와 같은 배지 체계를 사용한다. 최소 `수집완료`, `정정공고`, `취소공고`, `최초공고` 등은 같은 의미로 해석되게 한다.
- SQLModel과 sample 데이터 모두 `status`, `status_variant`, `version_label`, `version_variant`, `contract_name` 필드를 공통으로 사용해 결과 화면 표시 계약을 맞춘다.

체크리스트:

- [ ] 단계별 timeline badge/card 표현 규칙 정리
- [ ] 결과 화면의 낙찰업체/금액/계약일자 강조 방식 보강
- [ ] history와 timeline 역할 분리
- [ ] 상태 배지와 stage label 문구 기준 통일
- [ ] 정정/취소 이력과 버전 흐름을 timeline/history에 어떻게 노출할지 정리

완료 기준:

- 타임라인과 결과 화면만으로 입찰 이후 흐름을 해석할 수 있다.
- 사용자가 변경 이력과 현재 유효 버전을 혼동하지 않는다.

### Track E. 사전 탐색/사후 분석 초안

목표: 메인 운영 화면 외에 보조 탐색 화면의 최소 형태를 확보한다.

체크리스트:

- [ ] 사전 탐색 화면의 필수 지표/목록 정의
- [ ] 사후 분석 화면의 필수 지표/목록 정의
- [ ] 기존 repository/service 재사용 범위 확인
- [ ] 초기 화면은 placeholder가 아닌 실제 조회 데이터로 연결
- [ ] 측정 가능한 운영 지표와 이상 탐지 후보 지표 정리

완료 기준:

- 실무자가 보조 화면에서 탐색과 분석의 기본 흐름을 시연할 수 있다.

### Track F. 내보내기와 검증

목표: 화면 결과를 외부 공유와 운영 검증에 활용할 수 있게 한다.

수동 검증 시나리오 초안:

- 메인 리스트에서 검색어, 상태, 기관, 기간, 관심 여부 필터를 조합하고 정렬/페이지 크기/페이지네이션이 query param과 함께 유지되는지 확인한다.
- 리스트 행을 눌러 Drawer를 열고 `overview -> qualification -> attachments -> timeline -> history` 순서, 버전 라벨, 최신 유효본 안내, fallback 문구를 확인한다.
- 정정공고/취소공고 케이스에서 timeline의 사건 흐름과 history의 변경 요약이 분리되어 보이는지 확인한다.
- 리스트와 Drawer에서 관심 토글, 내부 상태 변경을 수행하고 화면이 partial 갱신으로 즉시 반영되는지 확인한다.
- Drawer에서 `상세 재실행`, `계약과정 재실행`, `크롤링 재실행`을 실행하고 성공/실패 alert와 `/operations` 링크가 함께 노출되는지 확인한다.
- `/operations`에서 `bid_detail_enrichment`, `contract_process_sync`, `bid_page_crawl` 로그를 필터링해 최근 실패 시각, 실패 단계, 요약 메시지, 재시도 여부를 판독할 수 있는지 확인한다.
- `/results`에서 상태 배지, 버전 배지, 낙찰업체, 낙찰금액, 계약금액, 계약일자가 한 화면에서 읽히는지 확인한다.

체크리스트:

- [ ] Excel 내보내기 범위와 컬럼 정의
- [ ] 주요 라우트/partial 회귀 테스트 추가
- [ ] 수동 검증 시나리오 정리
- [ ] 관련 문서와 실행 메모 동기화
- [ ] 버전 공고, 취소공고, 크롤링 실패 상태가 검증 시나리오에 포함되도록 정리

완료 기준:

- 핵심 화면 회귀 없이 내보내기와 운영 검증 절차를 재현할 수 있다.

## 5. 권장 구현 순서

1. Track A에서 메인 리스트와 필터 계약을 먼저 고정한다.
2. Track B에서 상세 Drawer 정보 구조를 재정리한다.
3. Track C에서 관심/상태/수동 액션을 연결한다.
4. Track D에서 timeline/results 시각화를 마무리한다.
5. Track E, F에서 보조 화면, 내보내기, 검증 절차를 정리한다.

보충 결정:

- 화면 우선순위는 `메인 리스트 -> 상세 Drawer -> /results -> /operations -> 보조 화면` 순서로 둔다.
- Phase 3의 핵심 사용자 흐름은 `리스트에서 후보 선별 -> Drawer에서 상세 검토 -> 필요 시 관심/상태/재실행 액션 수행 -> 결과 화면에서 후속 상태 확인`으로 정의한다.
- `메인 리스트`는 탐색과 선별의 진입점, `상세 Drawer`는 단건 검토 중심, `/results`는 계약과정통합공개와 후속 결과 판독, `/operations`는 운영 추적 화면으로 역할을 분리한다.
- 조회 화면과 업무 액션의 경계는 `조회 우선, 액션 보조` 원칙으로 둔다. 리스트에서는 `관심 토글`, `내부 상태 빠른 변경` 중심의 경량 액션만 허용하고, `detail/contract/crawl 재실행`은 상세 Drawer 내부에서만 제공한다.
- 수동 재실행 액션은 모두 `/operations`와 연결되는 운영 액션으로 간주한다. 현재 화면에서는 성공/실패 요약만 즉시 보여주고, 상세 추적은 `/operations`에서 확인하게 한다.
- 정정공고/취소공고 버전 표시는 메인 리스트, 상세 Drawer, timeline/history, `/results`까지 일관되게 적용하되, 기본 리스트는 `최신 유효본` 중심으로 유지한다.

## 6. 첫 구현 범위

Phase 3의 첫 구현 범위는 `정정공고/취소공고 버전 UX`로 잡는다.

우선순위 이유:

- 버전 공고 해석 규칙이 먼저 고정돼야 메인 리스트, 상세 Drawer, timeline, results 화면의 표시 계약이 흔들리지 않는다.
- 데이터 그리드 도입 여부는 그 다음 단계에서 상호작용 개선으로 붙여도 되지만, 버전 표시는 화면 의미 자체를 결정한다.
- 실제 실무에서는 최신 유효 공고와 이전 정정본을 혼동하지 않는 것이 검색 편의보다 더 중요하다.

첫 구현 체크리스트:

- [ ] 원공고, 정정본, 취소본, 최신 유효본의 화면 용어 확정
- [ ] 메인 리스트에서 최신본/이전본/취소 상태 표시 규칙 확정
- [ ] 상세 Drawer에서 원공고 관계와 버전 이력 노출 위치 확정
- [ ] timeline/history에 정정/취소 이벤트 표시 방식 확정
- [ ] 테스트 픽스처에 정정공고/취소공고 시나리오 추가

현재 반영 사항:

- `bid_version_changes` 테이블 초안 및 `sync_bid_change_history` CLI 초안이 추가됐다.
- `sync_phase2_batch`와 admin phase2 batch API에 change history sync 단계가 포함됐다.
- 상세 Drawer와 timeline/history는 `bid_version_changes` 기반 항목을 렌더링하도록 연결됐다.

다음 순서:

1. 실제 G2B 변경이력 응답 샘플로 필드명을 확정한다.
2. 현재 임시 버전 추론 로직을 정규화 필드 우선 로직으로 치환한다.
3. 그 다음 메인 리스트에 데이터 그리드 도입 여부를 검토한다.
4. 마지막으로 수동 액션과 운영 가시성을 붙인다.

## 7. 위험 포인트

- 화면 요구사항이 넓어지면 메인 리스트 + 상세 Drawer 우선순위가 흐려질 수 있다.
- HTMX partial 범위가 커지면 상태 동기화와 테스트 비용이 급격히 커질 수 있다.
- Excel 내보내기 컬럼이 늘어나면 화면용 DTO와 export DTO가 섞일 수 있다.
- 수동 재수집 액션은 운영 편의성과 오남용 방지 사이의 기준이 필요하다.
- 나라장터 스타일을 과도하게 복제하면 레거시 UX까지 답습할 수 있다.
- 정정공고/취소공고의 데이터 버전 규칙이 불명확하면 화면과 저장소 해석이 엇갈릴 수 있다.

대응:

- 먼저 사용자 핵심 흐름을 고정하고 그 밖의 화면은 보조로 분리한다.
- partial 계약은 query param, 응답 범위, 테스트 케이스와 함께 정의한다.
- export 전용 mapper 또는 serializer를 분리한다.
- 수동 액션은 admin token과 운영 로그를 기본 전제로 둔다.
- 시각 톤은 G2B를 참고하되 상호작용은 데이터 그리드 중심으로 현대화한다.
- 버전 공고 전략은 `docs/data/` 문서와 함께 원공고/정정본/취소본의 저장 및 표시 규칙을 조기 고정한다.
