# PRD: Woorisoft G2B 입찰 관리 시스템

## 1. 프로젝트 개요

- **프로젝트명**: Woorisoft G2B 입찰 자동화 및 관리 대시보드
- **목적**: 나라장터 오픈 API와 Playwright 크롤링을 결합해 입찰 공고 수집부터 계약 추적까지의 전 과정을 통합 관리하고, 사내 입찰 전략 수립을 지원하는 대시보드를 구축한다.
- **대상 사용자**: Woorisoft 입찰 담당자, 실무 검토자, 의사결정자
- **문제 정의**:
  - 입찰 공고 확인을 위해 매일 반복적으로 나라장터를 수동 탐색해야 한다.
  - API만으로는 첨부파일, 본문 텍스트, 일부 상세 정보 확보가 어렵다.
  - 공고, 개찰, 계약까지의 진행 상태를 한 화면에서 추적하기 어렵다.

## 2. 목표 및 핵심 지표 (OKRs)

### 목표

- 입찰 정보 수집 시간을 90% 이상 단축한다.
- 주요 공고 누락률을 0%에 가깝게 줄인다.
- 실무자가 기존 나라장터 사용 방식과 유사한 흐름으로 빠르게 적응할 수 있도록 한다.

### 핵심 지표

- 일일 신규 공고 자동 수집 및 분류 성공률 99% 이상
- 상세 페이지 및 첨부파일 메타데이터 수집 성공률 95% 이상
- 관심 공고 등록 후 상태 변경 감지 지연 시간 1일 이내
- 신규 사용자 기준 기본 사용 흐름 적응 시간 1일 이내

## 3. 타겟 사용자 페르소나

### 입찰 담당자

- 매일 나라장터에 접속해 AI, 소프트웨어 개발, 클라우드, 자동화 관련 공고를 검색한다.
- 공고 내용을 검토하고 제안서 작성 여부와 우선순위를 판단한다.
- 첨부파일, 예산, 일정, 발주기관 이력을 빠르게 파악해야 한다.

### 의사결정자

- 전체 입찰 파이프라인과 진행 현황을 요약해서 보고받고 싶어 한다.
- 어떤 공고에 집중해야 하는지 근거 기반으로 판단하고 싶어 한다.
- 경쟁 강도, 예산 규모, 기관 유형 등 전략 수립에 필요한 데이터가 필요하다.

## 4. 제품 범위

### 포함 범위 (In Scope)

- 나라장터 API 기반 공고, 사전규격, 낙찰 정보 수집
- Playwright 기반 상세 페이지 및 첨부파일 메타데이터 보강 수집
- 입찰 공고 리스트, 상세 보기, 필터, 즐겨찾기, 재수집 기능
- 계약 단계까지 이어지는 타임라인 시각화
- Excel 내보내기 기능

### 제외 범위 (Out of Scope)

- 자동 투찰 또는 제안서 자동 제출 기능
- 공인인증서 자체 관리 솔루션 개발
- 외부 고객용 공개 서비스 제공
- 모바일 앱 네이티브 개발

## 5. 상세 기능 요구사항

### 5.1 데이터 수집 엔진 (Backend & Scraper)

- **API 동기화**: 조달청 오픈 API 11종을 활용해 입찰 공고, 사전규격, 낙찰 정보를 주기적으로 수집한다.
- **하이브리드 크롤링**: API로 제공되지 않는 첨부파일 목록, 공고문 본문 텍스트, 추가 상세 정보를 Playwright로 수집한다.
- **로그인 세션 유지**: Persistent Context를 활용해 로그인 상태를 유지하고 인증이 필요한 페이지 접근을 지원한다.
- **중복 방지 및 갱신 처리**: 공고번호 기준으로 중복 저장을 방지하고 최신 상태를 반영한다.
- **실패 재시도**: 네트워크 오류, 타임아웃, DOM 변경 시 재시도 정책과 실패 로그를 남긴다.

### 5.2 대시보드 및 UI (Frontend)

- **레거시 친화형 UI**: 실무자에게 익숙한 나라장터 스타일의 테이블 중심 레이아웃과 Blue-Grey 계열 컬러 시스템을 적용한다.
- **Bootstrap 기반 관리형 UI**: 표, 폼, 필터, 배지, 탭, 모달, Offcanvas 패턴을 빠르게 조합할 수 있도록 Bootstrap 5 기반으로 설계한다.
- **통합 타임라인**: 사전규격 -> 입찰 -> 개찰 -> 계약 흐름을 한 줄의 타임라인으로 시각화한다.
- **스마트 필터**: 키워드, 수요기관, 예산 구간, 마감일, 상태별 자동 필터링을 지원한다.
- **빠른 상세 보기**: 리스트에서 선택 시 우측 Drawer에서 상세 내용과 첨부파일 목록을 즉시 확인한다.
- **검색 기능**: 공고명, 기관명, 공고번호 기반 검색을 지원한다.

### 5.3 관리 기능

- **관심 공고 관리**: 즐겨찾기 또는 관심 상태 지정으로 검토 대상을 별도 관리한다.
- **수동 재수집**: 특정 공고를 즉시 다시 읽어와 최신 상태로 동기화한다.
- **데이터 내보내기**: 조건에 맞는 리스트를 Excel 형식으로 다운로드한다.
- **상태 관리**: 수집완료, 검토중, 관심, 투찰완료 등 내부 운영 상태를 부여한다.

## 6. 비기능 요구사항

### 보안

- 국내 리전 IP 기반 환경을 사용해 나라장터 접근 시 IP 차단 리스크를 최소화한다.
- Cloudflare Tunnel 및 Zero Trust를 통해 관리자 접근을 제한한다.
- 인증 정보와 세션 데이터는 암호화된 저장소 또는 환경 변수로 관리한다.

### 성능

- 일일 배치 수집은 업무 시작 전 완료되는 것을 목표로 한다.
- Playwright 실행 시 Headless 모드와 동시 실행 제한으로 CPU/RAM 점유를 제어한다.
- 리스트 화면의 첫 응답 시간은 일반 조회 기준 3초 이내를 목표로 한다.

### 운영성

- 수집 실패 로그와 처리 이력을 확인할 수 있어야 한다.
- 특정 공고의 최종 수집 시각, 마지막 변경 시각을 추적할 수 있어야 한다.
- 향후 PostgreSQL 기준 운영 전환이 가능하도록 데이터 모델을 설계한다.

### 규정 준수

- 공공데이터 활용 가이드라인과 대상 사이트 이용 약관을 준수한다.
- 과도한 요청 빈도나 비정상 접근으로 간주될 수 있는 동작을 제한한다.

## 7. 기술 스택

- **Backend**: Python 3.10+, FastAPI
- **Crawler**: Playwright (Async Mode)
- **Database**: PostgreSQL, JSONB 활용
- **Frontend**: Jinja2 Template, Bootstrap 5, HTMX
- **Security**: Cloudflare Tunnel, Cloudflare Zero Trust
- **Infrastructure**: 가성비 중심 VPS 또는 Oracle Cloud Seoul Region

## 8. 데이터 아키텍처

### 8.1 주요 엔터티

- **Bids**: 시스템의 운영 중심 마스터. 공고번호, 차수, 공고명, 공고기관, 수요기관, 공고일시, 마감일시, 예산, 상태코드
- **Bid_Details**: 공고 상세 본문, 원본 API 응답, 크롤링 보강 데이터
- **Attachments**: 일반 첨부파일 메타데이터, 다운로드 링크, 로컬 저장 경로, 수집 시각
- **Bid_License_Limits**: 공고별 면허제한 및 허용업종 정보
- **Bid_Participation_Regions**: 공고별 참가가능지역 정보
- **Bid_Purchase_Items**: 구매대상물품/세부품명/수량/단가/납품조건 정보
- **Bid_Base_Amounts**: 기초금액, 예비가격 범위, 보험료, A값 관련 정보
- **Bid_Change_Histories**: 공고 변경 전후값 및 변경 항목 이력
- **Contract_Process_Integrations**: 계약과정통합공개 원본 및 연결 키 허브
- **Contract_Process_Awards**: 낙찰자정보 정규화 데이터
- **Contract_Process_Contracts**: 계약정보 정규화 데이터
- **Timeline_Stage_Snapshots**: 화면용 단계별 타임라인 요약 데이터
- **User_Favorites**: 사용자별 관심 공고 매핑 및 메모

### 8.2 데이터 설계 원칙

- API 원본과 가공 데이터를 분리 저장해 추적 가능성을 확보한다.
- 공고번호를 기준 키로 사용하되 차수, 변경 이력까지 고려한 확장 가능 구조로 설계한다.
- 첨부파일은 메타데이터와 실제 저장 위치를 분리 관리한다.
- 검색과 필터 성능을 위해 마감일, 기관명, 상태 컬럼 인덱스를 고려한다.

### 8.3 아키텍처 요약

- **운영 중심 허브**: `입찰공고정보서비스`를 기준으로 내부 `Bids` 마스터를 생성한다.
- **타임라인 보강 허브**: `계약과정통합공개서비스`를 통해 사전규격 -> 입찰 -> 낙찰 -> 계약 흐름을 연결하고 보강한다.
- **결과 확정 레이어**: `낙찰정보서비스`, `계약정보서비스`로 결과와 계약 상태를 확정한다.
- **기준정보 레이어**: `업종 및 근거법규서비스`를 통해 입찰 자격 및 업종 판단을 보조한다.
- **연결 키 원칙**: `bid_id`를 내부 기본 키로 사용하되, `bidNtceNo`, `bfSpecRgstNo`, `orderPlanUntyNo`, `prcrmntReqNo`를 함께 관리해 API 간 연결성을 확보한다.

### 8.4 데이터 관계 요약

- `Bids`는 시스템의 최상위 운영 엔터티다.
- `Bid_Details`, `Attachments`, `Bid_License_Limits`, `Bid_Participation_Regions`, `Bid_Purchase_Items`, `Bid_Base_Amounts`, `Bid_Change_Histories`는 모두 `Bids`를 기준으로 확장된다.
- `Contract_Process_Integrations`는 `Bids`와 연결되는 보강 허브이며, 하위에 `Contract_Process_Awards`, `Contract_Process_Contracts`, `Timeline_Stage_Snapshots`를 가진다.
- 기준정보 테이블인 `Industry_Base_Law_Rules`는 직접 FK 연결보다 업종코드 및 참가자격 해석 로직에서 간접 참조하는 구조가 적절하다.

## 9. 화면 설계 (Information Architecture)

### 메인 화면

- 실시간 입찰 공고 리스트
- 상단 검색창 및 필터 영역
- 상태별 요약 카드 또는 카운트 영역

### 상세 화면

- 우측 Drawer 기반 상세 보기
- 공고 기본 정보, 본문 요약, 첨부파일 목록
- 수동 재수집, 관심 등록, 내보내기 액션 버튼

### 사전 탐색 화면

- 발주계획 및 사전규격 리스트
- 본공고 전환 가능성이 높은 항목 추적

### 사후 분석 화면

- 낙찰 및 계약 결과 조회
- 경쟁사/기관별 결과 분석
- 예산 대비 실제 낙찰 흐름 확인

## 10. 사용자 흐름

1. 시스템이 매일 API 및 크롤링 기반으로 신규 공고를 수집한다.
2. 담당자가 메인 화면에서 키워드와 기관 기준으로 공고를 필터링한다.
3. 상세 Drawer에서 첨부파일과 본문을 검토하고 관심 공고로 등록한다.
4. 상태 변화가 발생하면 타임라인과 내부 상태값이 갱신된다.
5. 필요 시 리스트를 Excel로 내보내거나 특정 공고를 수동 재수집한다.

## 11. 제약사항 및 고려사항

- 나라장터 사이트 구조 변경 시 크롤링 로직 유지보수가 필요하다.
- 인증 기반 페이지 접근은 세션 만료, 보안 정책 변경에 영향을 받을 수 있다.
- VPS 자원 제약에 따라 동시 크롤링 수를 제한해야 할 수 있다.
- 운영 초기에는 SQLite로 시작할 수 있으나, 정식 운영은 PostgreSQL 기준으로 전환하는 것이 바람직하다.

## 12. 성공 기준 및 수용 조건

- 일일 배치 실행 후 신규 공고가 누락 없이 저장된다.
- 공고 상세 보기에서 핵심 본문 및 첨부파일 목록이 정상 노출된다.
- 관심 공고 등록, 상태 변경, 수동 재수집, Excel 내보내기가 정상 동작한다.
- 실무자가 기존 나라장터 대비 더 빠르게 공고 탐색과 검토를 수행할 수 있다고 평가한다.

## 13. 로드맵

### Phase 1

- API 기반 리스트 수집
- 초기 DB 스키마 구성
- 기본 리스트 조회 기능 구현

### Phase 2

- Playwright 연동
- 상세 데이터 및 첨부파일 수집 자동화
- 실패 재시도 및 로그 체계 정비

### Phase 3

- 나라장터 스타일 UI 적용
- 실무자 피드백 반영
- 필터, 관심 공고, 타임라인 기능 고도화

### Phase 4

- Cloudflare 기반 외부 접속 보안 강화
- 운영 환경 배포
- 모니터링 및 장애 대응 체계 정비

## 14. 구현 메모

초기 개발 단계에서는 SQLite 기반 프로토타입으로 빠르게 검증할 수 있다. 다만 본 PRD의 운영 목표와 데이터 아키텍처는 PostgreSQL 중심으로 정의한다. 따라서 초기 코드 예시는 별도 기술 문서나 `docs/` 하위 구현 가이드로 분리하고, 본 문서에는 제품 요구사항 중심 내용만 유지하는 것이 적절하다.

- `docs/backend-architecture.md`: 수집 파이프라인, 스케줄러, 에러 처리 정책
- `docs/data/data-model.md`: 테이블 스키마, 인덱스, 마이그레이션 전략
- `docs/api/api-spec.md`: 내부 API 및 화면 연동 명세
