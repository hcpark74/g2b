# System Architecture: G2B Tender Management

## 1. 목적

- 본 문서는 Woorisoft G2B 입찰 관리 시스템의 외부 API 역할, 내부 데이터 흐름, 단계별 책임을 정리한 설계 문서다.
- 설계 기준은 `docs/PRD.md`, `docs/api/`, `docs/data/data-model.md`를 따른다.

## 2. 전체 구조

나라장터 입찰 프로세스는 아래 6개 영역으로 나눈다.

1. 사전 탐색
2. 본공고 및 입찰
3. 개찰 및 낙찰
4. 계약
5. 통합 추적
6. 표준화 및 거버넌스

핵심 원칙:

- 운영 중심 HUB는 `입찰공고정보서비스`
- 타임라인 중심 HUB는 `계약과정통합공개서비스`
- 단계별 상세 정보는 개별 API가 담당
- 내부 메인 엔터티는 `입찰공고(bid)`
- 연결 키는 `bidNtceNo`, `bfSpecRgstNo`, `orderPlanNo`, `prcrmntReqNo`를 함께 관리

## 3. 단계별 아키텍처

### 3.1 사전 탐색

목적:

- 본공고 이전에 사업 징후를 포착하고 선제적으로 검토한다.

주요 API:

- `발주계획현황서비스`
- `사전규격정보서비스`
- `조달요청서비스`

주요 활용:

- 연간/월간 발주 예정 사업 파악
- 사전규격 검토를 통한 기술 적합성 확인
- 조달요청 단계 추적으로 본공고 전환 가능성 모니터링

설계 메모:

- 이 단계는 아직 `bid_id`가 없을 수 있으므로 `orderPlanNo`, `bfSpecRgstNo`, `prcrmntReqNo` 중심으로 저장한다.
- 본공고 생성 전 리드 파이프라인 성격의 데이터다.

### 3.2 본공고 및 입찰

목적:

- 실제 입찰 참여 여부를 판단하고 메인 리스트를 운영한다.

주요 API:

- `입찰공고정보서비스`
- `업종 및 근거법규서비스`
- `가격정보현황서비스`
- `사용자정보 서비스`

주요 활용:

- 메인 대시보드 공고 리스트 생성
- 공고번호, 마감일, 기관, 금액, 상태 중심 운영
- 참여 자격 및 면허 검토 보조
- 투찰 전략 수립 보조

설계 메모:

- `입찰공고정보서비스`가 내부 시스템의 운영 중심 HUB다.
- `업종 및 근거법규서비스`는 참가 가능성 자동판단의 보조 기준이며, 최종 판단은 공고문 참가자격 문구와 함께 해석해야 한다.
- `가격정보현황서비스`와 `사용자정보 서비스`는 핵심 원천이라기보다 분석/보조 데이터 레이어에 가깝다.

### 3.3 개찰 및 낙찰

목적:

- 개찰 결과와 낙찰 패턴을 확인하고 다음 전략에 반영한다.

주요 API:

- `낙찰정보서비스`

주요 활용:

- 낙찰 업체 및 낙찰 금액 확인
- 낙찰률 분석
- 경쟁 구도와 입찰 패턴 분석

설계 메모:

- 낙찰 데이터는 결과 분석과 전략 보정의 핵심 소스다.
- 공고별 최종 결과를 내부 분석 테이블에 별도 정규화하는 것이 바람직하다.

### 3.4 계약

목적:

- 낙찰 이후 실제 계약 체결 정보를 확인한다.

주요 API:

- `계약정보서비스`

주요 활용:

- 계약번호, 계약명, 계약금액, 계약일자 확정
- 계약기관/수요기관 기준 분석

설계 메모:

- 낙찰과 계약은 동일하지 않으므로 별도 단계로 분리해야 한다.
- 계약 미존재는 오류가 아니라 미체결 상태일 수 있다.

### 3.5 통합 추적

목적:

- 특정 식별자 하나로 사전규격부터 계약까지 전체 흐름을 타임라인으로 연결한다.

주요 API:

- `계약과정통합공개서비스`

주요 활용:

- 상세 화면 타임라인 구성
- 단계 간 연결 키 보강
- 사전규격 -> 입찰 -> 낙찰 -> 계약 흐름 재구성

설계 메모:

- 이 API는 전체의 단일 원천이라기보다 타임라인/연결 보강용 HUB다.
- `bidwinrInfoList`, `cntrctInfoList`는 별도 정규화가 필요하다.
- 빈 낙찰/계약 데이터는 상태 미도달 가능성을 우선 고려한다.

### 3.6 표준화 및 거버넌스

목적:

- 수집 데이터를 내부 기준에 맞춰 정제하고 재사용 가능하게 관리한다.

주요 참고 서비스:

- `공공데이터개방표준서비스`

주요 활용:

- 공통 필드명 정리
- 내부 DB 적재 규칙 정립
- 분석 및 내보내기 포맷 일관성 유지

설계 메모:

- 이 영역은 업무 조회 단계가 아니라 데이터 관리 정책 영역으로 보는 것이 적절하다.

## 4. 데이터 흐름

현재 운영 구현은 전체 공고를 내부 DB에 적재한 뒤 검색하는 흐름을 일부 포함하지만, 목표 방향은 `외부 API 검색 + 관심 공고 중심 영속화`다. 상세 전환안은 `docs/architecture/favorite-centric-search-architecture.md`를 따른다.

```text
[발주계획/사전규격/조달요청]
        ->
[입찰공고정보서비스]
        ->
[낙찰정보서비스]
        ->
[계약정보서비스]
        ->
[내부 정규화 DB]

                     \-> [계약과정통합공개서비스] -> [타임라인 보강]
                     \-> [업종 및 근거법규서비스] -> [자격 검토 보강]
                     \-> [가격/사용자정보 서비스] -> [전략 분석 보강]
```

## 5. 내부 데이터 책임 분리

### 운영 중심 데이터

- `bids`
- `bid_details`
- `attachments`

### 기준정보 데이터

- `industry_base_law_rules`
- `industry_inclusion_licenses`

### 통합 추적 데이터

- `contract_process_integrations`
- `contract_process_awards`
- `contract_process_contracts`
- `timeline_stage_snapshots`

### 운영 로그 데이터

- `sync_jobs`
- `timeline_logs`

## 6. 키 설계 원칙

### 1차 키

- `bid_id = {bidNtceNo}-{bidNtceOrd}`

### 보조 연결 키

- `bidNtceNo`
- `bfSpecRgstNo`
- `orderPlanNo`
- `orderPlanUntyNo`
- `prcrmntReqNo`

설계 메모:

- 내부 중심 키는 `bid_id`지만, 외부 API 연결은 보조 키 다중 관리가 필수다.
- 본공고 이전 데이터는 `bid_id`가 없을 수 있으므로 보조 키 기반 조인이 가능해야 한다.

## 7. 권장 API 우선순위

### 운영 우선순위

1. `입찰공고정보서비스`
2. `계약과정통합공개서비스`
3. `낙찰정보서비스`
4. `계약정보서비스`
5. `사전규격정보서비스`
6. `발주계획현황서비스`
7. `조달요청서비스`
8. `업종 및 근거법규서비스`
9. `가격정보현황서비스`
10. `사용자정보 서비스`

### 이유

- 실제 운영 화면의 중심은 입찰공고 리스트다.
- 통합 타임라인과 연결 보강에는 계약과정통합공개서비스가 중요하다.
- 결과/계약 데이터는 단계별 상태 확정에 필수다.
- 업종/가격/사용자 정보는 전략 보조 레이어다.

## 8. 아키텍처 결론

- 현재 구상한 `준비 -> 공고 -> 결과 -> 계약 -> 통합 관리` 구조는 방향이 맞다.
- 다만 시스템 설계에서는 `계약과정통합공개서비스`를 전체 원장으로 보기보다 `타임라인 연결 허브`로 정의하는 것이 정확하다.
- 실제 운영 허브는 `입찰공고정보서비스`, 타임라인 허브는 `계약과정통합공개서비스`로 구분하는 것이 가장 안정적이다.

## 9. 현재 코드 아키텍처 반영

현재 코드베이스는 아래 흐름으로 정리되어 있다.

```text
sample_data
  -> repository
  -> service
  -> presentation mapper
  -> view-model
  -> Jinja2 template
```

레이어별 역할:

- `sample_data`: 초기 UI/라우팅 검증용 샘플 원본 데이터
- `repositories`: 향후 SQLite/PostgreSQL 조회로 교체 가능한 데이터 접근 계층
- `services`: 화면이나 기능 단위 조회 조립 계층
- `presentation/mappers`: 템플릿 친화 데이터 구조로 변환
- `presentation/viewmodels`: 화면별 전용 모델
- `templates`: Bootstrap 5 + HTMX 기반 렌더링 계층

설계 의미:

- 현재는 샘플 데이터 기반이지만, DB 연결 시 repository 구현체만 바꾸면 상위 계층을 최대한 유지할 수 있다.
- 프론트는 원본 API 응답이 아니라 view-model 기준으로 렌더링되도록 정리되어 있다.
