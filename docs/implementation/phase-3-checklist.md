## 1. 범위 확정

- [x] Phase 3 대상 화면 우선순위 확정
- [x] 메인 리스트/상세 Drawer/결과 화면 역할 분리 확정
- [x] 수동 액션과 조회 화면 경계 확정
- [x] 정정공고/취소공고 버전 표시 범위 확정

결정안:

- 화면 우선순위는 `메인 리스트 -> 상세 Drawer -> /results -> /operations -> 보조 화면` 순서로 둔다.
- Phase 3의 핵심 사용자 흐름은 `리스트에서 후보 선별 -> Drawer에서 상세 검토 -> 필요 시 관심/상태/재실행 액션 수행 -> 결과 화면에서 후속 상태 확인`으로 정의한다.
- `메인 리스트`는 탐색과 선별의 진입점으로 유지한다. 검색, 필터, 정렬, 버전 상태, 관심 여부를 빠르게 판독하는 역할에 집중한다.
- `상세 Drawer`는 단건 검토의 중심 화면으로 유지한다. overview, 자격, 첨부, timeline, history를 한 문맥에서 확인하고 관심/상태/재실행 같은 경량 액션을 수행하는 지점으로 본다.
- `/results`는 계약과정통합공개와 후속 결과 판독에 집중한다. 메인 리스트의 상세 대체물이 아니라 낙찰업체, 낙찰금액, 계약일자, 진행 결과를 밀도 있게 확인하는 별도 결과 화면으로 둔다.
- `/operations`는 운영 추적 화면으로 유지한다. 수동 액션과 배치 실행의 성공/실패, 재시도, 에러 메시지, 최근 실행 이력을 확인하는 역할에 집중한다.
- 조회 화면과 업무 액션의 경계는 `조회 우선, 액션 보조` 원칙으로 둔다. 기본 화면은 탐색과 판독을 우선하고, 액션은 현재 문맥을 해치지 않는 범위에서만 배치한다.
- 리스트에서는 `관심 토글`, `내부 상태 빠른 변경` 같은 즉시성 높은 액션만 허용한다. `detail/contract/crawl 재실행`은 행 단위 대량 액션으로 확장하지 않고 상세 Drawer 내부에서만 제공한다.
- 수동 재실행 액션은 모두 `/operations`와 연결되는 운영 액션으로 간주한다. 실행 직후에는 현재 화면에서 성공/실패 요약을 보여주고, 상세 추적은 `/operations`에서 확인하게 한다.
- 정정공고/취소공고 버전 표시는 메인 리스트, 상세 Drawer, timeline/history, `/results`까지 일관되게 적용한다. 다만 표시 밀도는 화면 역할에 맞춰 다르게 둔다.
- 메인 리스트 기본값은 `bid_no` 기준 `최신 유효본`만 노출한다. 사용자가 원할 때만 과거 차수 또는 취소본을 볼 수 있는 필터를 추가하는 방향으로 간다.
- 상세 Drawer는 현재 공고를 `원공고`, `정정본`, `취소본` 중 하나로 명시하고, 같은 `bid_no` 그룹의 버전 흐름과 최신 유효본 여부를 함께 보여준다.
- timeline은 사건 흐름을, history는 화면용 요약 이력을 담당한다. 정정 등록, 취소 등록, 최신 유효본 변경 같은 버전 이벤트는 timeline에 우선 반영하고, history에는 요약 카드나 목록으로 재정리한다.
- `/results`는 현재 보고 있는 공고가 최신 유효본인지, 과거 버전인지, 취소본인지 식별 가능한 상태 배지를 유지한다. 결과 데이터가 특정 버전에 귀속된다는 점을 혼동하지 않도록 버전 라벨을 함께 노출한다.

## 2. 메인 리스트/필터

- [x] 검색어, 상태, 기관, 기간, 관심 여부 필터 정리
- [x] 정렬/페이지 크기/총 건수 UX 정리
- [x] 테이블 컬럼 우선순위와 스크롤 정책 정리
- [x] 데이터 그리드 도입 여부와 HTMX 연동 경계 정리
- [x] empty state와 로딩 표현 정리

결정안:

- 메인 리스트의 기본 목적은 `빠른 후보 선별`로 둔다. 한 화면에서 검색, 필터, 정렬, 버전 상태, 관심 여부를 먼저 판독하고 상세 검토는 Drawer로 넘긴다.
- 기본 정렬은 `마감임박 우선`보다 `업데이트 최신순`에 둔다. 실무 운영 기준으로 신규 수집, 정정, 취소, 상태 변화가 먼저 보여야 하기 때문이다.
- 보조 정렬 옵션은 `마감일 빠른순`, `마감일 늦은순`, `공고일 최신순`, `기관명`, `입찰명` 정도로 제한한다. 1차 Phase 3에서는 다중 정렬보다 단일 정렬 전환을 우선한다.
- 페이지 크기 기본값은 `25`로 둔다. 선택 옵션은 `25 / 50 / 100`으로 제한해 테이블 밀도와 응답 비용을 함께 관리한다.
- 총 건수는 리스트 상단에 `현재 조건 결과 n건` 형태로 항상 노출한다. 페이지 이동 전에도 필터 효과를 바로 확인할 수 있어야 한다.
- 필터와 정렬, 페이지 크기는 모두 URL query param으로 유지한다. 새로고침, 공유, partial 갱신 이후에도 현재 상태를 복원할 수 있어야 한다.
- 컬럼 우선순위는 `공고명`, `기관`, `상태`, `버전 상태`, `마감일`, `관심 여부`, `내부 상태`, `최종 갱신 시각` 순으로 둔다. 이 우선 컬럼만으로 1차 선별이 가능해야 한다.
- `공고명`과 `기관`은 좌측 고정 우선 컬럼으로 본다. 나머지 보조 메타데이터는 가로 스크롤 영역으로 보내고, 모바일/좁은 화면에서는 일부 컬럼을 축약한다.
- 테이블은 `행 클릭 시 Drawer 열기`를 기본 상호작용으로 유지한다. 인라인 확장행보다 우측 검토 문맥을 유지하는 쪽을 우선한다.
- 가로 스크롤은 허용하되, 컬럼 수를 무리하게 늘리지 않는다. 1차 화면은 `핵심 판독 컬럼 + 보조 컬럼` 구조로 나누고, 첨부 수나 세부 자격처럼 Drawer에서 보는 정보는 리스트에서 제외한다.
- 데이터 그리드는 Phase 3에서 `즉시 전면 도입`보다 `현재 HTMX 테이블 계약을 유지한 점진 강화`를 기본안으로 둔다. 먼저 서버 정렬/query 계약과 컬럼 정책을 고정하고, 이후 필요 시 AG Grid나 Tabulator 검토를 붙인다.
- HTMX 연동 경계는 `리스트 본문, 페이지네이션, 정렬, 필터 결과 영역`까지로 제한한다. 행 선택에 따른 상세 조회는 Drawer partial로 분리하고, 결과적으로 `목록 partial`과 `Drawer partial`을 별도 계약으로 유지한다.
- query param 계약에는 최소 `q`, `status`, `agency`, `date_from`, `date_to`, `favorite`, `sort`, `page`, `page_size`, `include_versions`를 포함한다.
- `include_versions` 기본값은 `false`로 둔다. 기본 리스트는 `최신 유효본` 중심으로 유지하고, 버전 포함 조회는 명시적 탐색 모드로 분리한다.
- empty state는 조건에 따라 문구를 나눈다. 데이터가 전혀 없을 때는 `표시할 공고가 없습니다.`, 필터 결과가 없을 때는 `조건에 맞는 공고가 없습니다. 필터를 조정해보세요.`처럼 구분한다.
- 로딩 표현은 전체 페이지 스피너보다 `리스트 영역 skeleton 또는 로딩 행`을 우선한다. 필터 변경 시 현재 화면 문맥을 유지하면서 갱신 중임을 보여주는 방식이 더 적합하다.
- 정렬 변경, 필터 적용, 페이지 이동 후에는 리스트 상단 요약과 테이블만 갱신하고, 이미 열린 Drawer는 사용자가 다른 행을 선택할 때까지 유지하는 방향을 우선 검토한다. 다만 현재 선택한 공고가 필터 결과에서 사라지면 Drawer는 닫거나 `현재 목록 조건에서는 제외된 공고` 안내를 준다.

## 3. 상세 Drawer

- [x] overview/qualification/attachments/timeline/history 섹션 재정리
- [x] 크롤링 본문/요약 표시 규칙 정리
- [x] 계약과정통합공개 결과 강조 방식 정리
- [x] placeholder fallback 정리
- [x] 정정/취소/최신본 상태 표시 규칙 정리

결정안:

- 상세 Drawer의 역할은 `단건 검토의 중심 화면`으로 고정한다. 사용자가 메인 리스트를 떠나지 않고도 공고 판단에 필요한 핵심 맥락을 한 번에 읽을 수 있어야 한다.
- 섹션 순서는 `overview -> qualification -> attachments -> timeline -> history`로 둔다. 먼저 현재 공고의 핵심 요약을 보여주고, 그다음 검토 조건, 첨부, 사건 흐름, 요약 이력을 따라가게 한다.
- `overview`는 가장 먼저 읽히는 요약 영역으로 사용한다. 공고명, 기관, 공고번호/차수, 마감일, 계약 방식, 예산 또는 추정금액, 내부 상태, 관심 여부, 버전 상태를 우선 배치한다.
- `overview` 상단에는 상태 배지를 모아 보여준다. 최소 `원공고/정정본/취소본`, `최신 유효본 여부`, 내부 상태, 관심 여부가 즉시 보이도록 한다.
- `qualification`은 실무 검토에서 바로 판단해야 하는 항목 위주로 압축한다. 참가 자격, 지역 제한, 업종/면허, 공동수급, 실적 제한처럼 입찰 가능 여부에 직접 영향을 주는 정보가 먼저 와야 한다.
- `attachments`는 파일명 나열보다 검토 가능성을 우선한다. 첨부 개수, 주요 파일명, 파일 유형 또는 출처 메타데이터를 먼저 보여주고, 세부 목록은 그 아래에서 확인하게 한다.
- `timeline`은 사건 흐름 중심으로 유지한다. 계약 단계와 함께 `정정 등록`, `취소 등록`, `최신 유효본 변경` 같은 버전 이벤트를 시간순으로 보여주는 영역으로 정의한다.
- `history`는 화면용 요약 이력으로 유지한다. timeline의 원시 사건 로그를 반복하지 않고, 버전별 핵심 차이, 계약 결과 요약, 최근 변경 요약을 카드나 목록으로 재정리한다.
- 크롤링 본문은 `overview`의 하위 요약 블록 또는 별도 접힘 영역으로 노출하되, 본문이 섹션 구조를 압도하지 않게 한다. 기본은 `본문 요약 우선`, 원문성 텍스트는 보조로 둔다.
- `crawl_data.text_summary`가 있으면 이를 `본문 요약`으로 우선 노출한다. 사용자는 긴 본문보다 먼저 핵심 추출 내용을 읽을 수 있어야 한다.
- 크롤링 원문 또는 장문 추출 텍스트는 요약 아래의 접힘 영역으로 둔다. 원문이 없으면 섹션 전체를 과도하게 비우기보다 `본문 추출 데이터 없음` 안내로 간단히 처리한다.
- 계약과정통합공개 결과는 `overview` 안의 강조 카드 또는 상단 보조 패널로 노출한다. 낙찰업체, 낙찰금액, 계약일자, 계약명처럼 결과 판독에 핵심인 값은 timeline/history보다 먼저 읽히게 한다.
- 결과 데이터가 있으면 `계약 결과 있음` 상태를 상단에서 바로 인지할 수 있게 하고, 상세 값은 같은 문맥에서 확인하게 한다. 결과 데이터가 없을 때는 빈 카드 대신 섹션을 축약하거나 `아직 결과 데이터 없음` 수준으로 처리한다.
- placeholder는 `구조 유지용`으로만 제한한다. 실제 데이터가 없는 경우에도 과장된 더미 값은 쓰지 않고, `미수집`, `정보 없음`, `아직 반영 전` 같은 명확한 상태 문구를 사용한다.
- fallback 문구는 데이터 부재 이유를 구분한다. 수집 전이면 `아직 수집되지 않음`, 소스상 값이 없으면 `제공 정보 없음`, 필드 매핑 전이면 `표시 규칙 정리 중`처럼 분리하는 방향을 유지한다.
- 정정/취소/최신본 상태 표시는 Drawer 헤더와 overview에 중복 배치해도 괜찮다. 사용자가 스크롤 위치와 무관하게 현재 공고의 버전 의미를 잃지 않도록 하는 것이 더 중요하다.
- 현재 공고가 최신 유효본이 아니면 `이 공고는 과거 버전입니다` 또는 `이 공고는 취소본입니다` 같은 강한 안내를 상단에 둔다. 동시에 `최신 유효본` 또는 관련 버전으로 이동할 수 있는 링크/버튼을 후속 구현 후보로 남긴다.
- 같은 `bid_no` 아래 다른 차수가 있으면 버전 흐름을 최소한 `차수 목록 + 현재 선택 + 최신 유효본 표시`로 보여준다. 이 정보는 overview 하단 또는 history 상단에 배치하는 것을 기본안으로 둔다.
- Drawer는 모든 정보를 다 보여주기보다 `판단 우선순위`에 따라 접고 펴는 구조를 택한다. overview와 qualification은 기본 펼침, attachments/timeline/history는 상황에 따라 접힘 허용이 적합하다.

- 실무 시나리오 1차 범위는 `사전 탐색에서 발주계획/사전규격/조달요청 후보를 찾고 -> 본공고 보기 또는 제목 클릭으로 메인 리스트 검색으로 넘어가고 -> 동일 조건 공고 목록을 검토하거나 CSV로 내보내기` 흐름으로 고정한다.
- `/prespecs`는 실제 조회, 필터, 결과 건수, 연결 상태 배지, 본공고 이동 링크를 지원하고, `/api/v1/bids/export`는 현재 조건 기준 CSV 내보내기를 지원한다.
