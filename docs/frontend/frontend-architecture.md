# Frontend Architecture

## 1. 목적

- 본 문서는 Woorisoft G2B 입찰 관리 시스템의 프론트엔드 정보 구조, 화면 구성, 상호작용 원칙을 정의한다.
- 백엔드 설계가 `입찰공고정보서비스` 중심 허브로 정리된 상태를 전제로, 프론트도 `공고 리스트 중심 + 우측 상세 패널 + 단계별 보조 화면` 구조로 설계한다.
- UI 프레임워크는 `Bootstrap 5 + Jinja2 + HTMX` 조합을 기본으로 한다.

## 2. 프론트엔드 설계 원칙

- 실무자가 익숙한 나라장터 계열 UX를 유지하되, 정보 밀도와 탐색 속도는 개선한다.
- 첫 진입점은 항상 `입찰공고 리스트`다.
- 상세는 별도 페이지보다 `우측 Drawer`를 기본 패턴으로 사용한다.
- 단계별 추가 정보는 리스트를 떠나지 않고 점진적으로 확장 확인할 수 있어야 한다.
- UI는 화려함보다 업무 집중도, 비교 가능성, 빠른 판독성을 우선한다.
- Bootstrap 기본 컴포넌트를 우선 활용하고, 시각 톤만 나라장터 친화형으로 재조정한다.

## 3. 전체 화면 구조

### 3.1 글로벌 레이아웃

```text
Top Header
Left Navigation
Main Content Area
Right Drawer (Contextual Detail)
```

구성:

- `Top Header`: 서비스명, 마지막 동기화 시각, 빠른 검색, 사용자 메뉴
- `Left Navigation`: 메인 화면, 사전 탐색, 사후 분석, 관심 공고, 운영 로그
- `Main Content Area`: 리스트/분석/검색 결과 본문
- `Right Drawer`: 선택된 공고의 상세 정보, 첨부, 타임라인, 액션 버튼

### 3.2 기본 화면 흐름

1. 사용자가 메인 리스트에 진입한다.
2. 상단 필터와 검색으로 공고를 좁힌다.
3. 테이블의 행을 선택하면 우측 Drawer가 열린다.
4. Drawer에서 첨부파일, 자격, 타임라인, 관련 액션을 처리한다.
5. 더 깊은 비교/분석이 필요하면 사후 분석 또는 사전 탐색 화면으로 이동한다.

## 4. 정보 구조 (IA)

### 4.1 메인 네비게이션

- `입찰 공고`
- `사전 탐색`
- `사후 분석`
- `관심 공고`
- `운영 현황`

### 4.2 화면별 역할

#### `입찰 공고`

- 메인 운영 화면
- 신규/변경 공고 확인
- 상태 관리
- 상세 Drawer 진입 중심

#### `사전 탐색`

- 발주계획, 사전규격, 조달요청 기반 선행 리드 확인
- 본공고 전환 가능 항목 추적

#### `사후 분석`

- 낙찰/계약 결과 분석
- 낙찰률, 기관별 패턴, 경쟁 구도 확인

#### `관심 공고`

- 실무 검토 중인 공고만 모아 집중 관리
- 메모, 상태, 마감 임박 우선 확인

#### `운영 현황`

- 최근 동기화 상태
- 수집 실패 내역
- 수동 재수집 결과

## 5. 핵심 화면 설계

### 5.1 메인 입찰 공고 화면

#### 상단 영역

- 페이지 제목
- 마지막 동기화 시각
- 빠른 검색 입력창
- 수동 동기화 버튼

#### 요약 영역

- `신규 공고`
- `오늘 마감`
- `관심 공고`
- `상태 변경 감지`

설계 메모:

- 카드 수는 4개 전후로 제한하고, 숫자와 기준 시점을 함께 표시한다.

#### 필터 영역

- 키워드
- 수요기관
- 공고기관
- 상태
- 업무구분(용역/물품/공사/외자/기타)
- 마감일 범위
- 예산 범위
- 업종 제한 여부

설계 메모:

- 자주 쓰는 필터는 항상 노출, 드문 필터는 펼침 패널에 배치한다.

#### 리스트 영역

권장 컬럼:

- 관심 여부
- 상태
- 업무구분
- 공고번호
- 공고명
- 공고기관
- 수요기관
- 계약체결방법
- 추정가격
- 공고일시
- 마감일시
- 개찰일시
- 변경 여부

설계 메모:

- 나라장터 스타일을 유지하되, 긴 공고명은 2줄까지 허용하고 핵심 날짜 컬럼은 고정 폭으로 유지한다.
- 행 클릭 전체를 Drawer 오픈 액션으로 사용한다.

### 5.2 공고 상세 Drawer

#### 상단 헤더

- 공고명
- 상태 배지
- 즐겨찾기 버튼
- 다시 읽어오기 버튼
- 닫기 버튼

#### 탭 구성 권장

- `개요`
- `자격/제한`
- `첨부파일`
- `타임라인`
- `이력`

#### `개요` 탭

- 공고번호/차수
- 공고기관/수요기관
- 공고일/마감일/개찰일
- 계약체결방법
- 예산/추정가격/VAT
- 공고 상세 링크

#### `자격/제한` 탭

- 업종제한 여부
- 면허제한 목록
- 허용업종 목록
- 참가가능지역
- 공동수급 방식
- 참가자격 요약 문구

#### `첨부파일` 탭

- 일반 공고 첨부
- e발주 첨부
- 파일명, 유형, 출처, 다운로드 액션

#### `타임라인` 탭

- 사전규격
- 입찰공고
- 개찰/낙찰
- 계약

설계 메모:

- 타임라인은 세로형보다 가로형 단계 표시 + 이벤트 리스트 혼합 구조가 실무 가독성이 높다.

#### `이력` 탭

- 변경일시
- 변경 항목
- 변경 전값
- 변경 후값

### 5.3 리스트 확장행 상세 패턴

나라장터 통합 검색 결과 화면처럼, 일부 화면에서는 우측 Drawer와 별도로 `행 하단 확장 상세` 패턴을 사용할 수 있다.

권장 적용 화면:

- 사전 탐색 화면
- 통합 검색 결과 화면
- 단계 진행 비교가 중요한 대량 조회 화면

구성:

- 기본 행: 사업명, 사업번호, 기관, 일자, 업무구분 등 요약 정보
- 확장행: 단계 카드형 타임라인
- 상단 상태: `진행`, `미진행`, 업무구분 태그

설계 메모:

- 메인 운영 화면은 `우측 Drawer`를 기본으로 유지한다.
- 다건 비교가 중요한 화면은 `행 확장형 타임라인`이 더 효율적일 수 있다.
- 따라서 상세 패턴은 `Drawer 기본 + 확장행 보조` 구조가 적절하다.

### 5.4 사전 탐색 화면

구성:

- 상단 검색/기간 필터
- 발주계획 리스트
- 사전규격 리스트
- 조달요청 상태 리스트
- 본공고 연결 여부 컬럼

핵심 목적:

- 아직 공고가 뜨지 않았지만 추적 가치가 있는 건을 미리 발견

### 5.5 사후 분석 화면

구성:

- 기간/기관/업무구분 필터
- 낙찰률 분포 요약
- 낙찰 업체 리스트
- 계약 체결 현황 테이블
- 공고 대비 계약 확정 여부 비교

핵심 목적:

- 결과 데이터를 전략 학습에 재사용

## 6. 컴포넌트 전략

### 필수 공통 컴포넌트

- `PageHeader`
- `SummaryStatCard`
- `FilterBar`
- `BidTable`
- `StatusBadge`
- `FavoriteToggle`
- `DetailDrawer`
- `AttachmentList`
- `TimelinePanel`
- `ChangeHistoryTable`
- `EmptyState`
- `SyncStatusBanner`

Bootstrap 매핑 예시:

- `SummaryStatCard` -> `card`
- `FilterBar` -> `form`, `input-group`, `accordion`
- `BidTable` -> `table`, `table-hover`
- `StatusBadge` -> `badge`
- `DetailDrawer` -> `offcanvas` 또는 커스텀 right drawer
- `TimelinePanel` -> `nav`, `progress`, `list-group` 조합

### 설계 원칙

- 컴포넌트는 시각적 재사용보다 데이터 역할 중심으로 분리한다.
- 상태 배지, 날짜 표시, 금액 포맷은 공통 포맷터를 사용한다.
- 테이블 셀은 단순 문자열보다 `기관`, `상태`, `마감 임박`, `첨부 존재` 같은 업무 신호를 전달해야 한다.

## 7. 상호작용 원칙

### 기본 상호작용

- 리스트 행 클릭 -> Drawer 오픈
- 상태 배지 클릭 -> 상태 필터 적용
- 즐겨찾기 클릭 -> 즉시 토글, 리스트 유지
- 다시 읽어오기 클릭 -> 비동기 처리 후 상태 갱신

### 피드백 원칙

- 수집/재수집은 토스트 + 행 단위 상태 반영
- 대량 새로고침보다 변경된 행만 부분 갱신
- 실패 시 기술 오류보다 사용자 행동 중심 메시지로 전달

예:

- `재수집 요청이 접수되었습니다.`
- `첨부파일 목록을 다시 불러오지 못했습니다.`

## 8. 시각적 방향

### 스타일 방향

- 나라장터 레거시 친화형
- 테이블 중심
- Blue-Grey 기반 중립색
- 중요 상태만 제한적으로 강조색 사용
- Bootstrap 관리자형 레이아웃 기반

### 색상 역할

- 기본 배경: 연한 회색 또는 아이보리 계열
- 테이블 헤더: desaturated blue-grey
- 링크/강조: 중간 톤 blue
- 경고: amber
- 성공/완료: muted green
- 변경/주의: orange-red

### 타이포그래피

- 기본은 가독성 높은 산세리프
- 숫자/공고번호/금액은 고정폭 계열 보조 사용 고려
- 제목보다 데이터 자체가 강조되도록 정보 밀도를 우선한다.

## 9. 모바일/반응형 전략

- 메인 대상은 데스크톱이지만 태블릿까지는 안정적으로 대응한다.
- 모바일에서는 좌측 네비게이션을 접고, Drawer를 전체 화면 시트로 전환한다.
- 테이블은 모바일에서 카드 리스트로 단순 변환하기보다 핵심 컬럼만 남긴 요약 리스트로 전환한다.

## 10. 기술 적용 메모

- `Jinja2 + Bootstrap 5 + HTMX` 기준으로 서버 렌더링된 리스트/Drawer 조각을 빠르게 갱신한다.
- 필터 변경 시 전체 페이지 리로드보다 리스트 부분만 부분 갱신한다.
- Drawer 내용은 별도 partial endpoint로 제공하는 것이 적절하다.
- Bootstrap 컴포넌트를 우선 사용하고, 과도한 커스텀 CSS는 지양한다.

권장 partial 예시:

- `GET /partials/bids/table`
- `GET /partials/bids/{bid_id}/drawer`
- `GET /partials/dashboard/summary`

## 11. 설계 결론

- 프론트의 시작점은 `입찰 공고 리스트`다.
- 상세는 `우측 Drawer`, 흐름은 `타임라인`, 판단은 `자격/첨부/이력` 탭으로 나누는 것이 가장 실무 친화적이다.
- 나라장터 스타일을 완전히 벗어나기보다, 익숙한 구조를 유지하면서 탐색 속도와 정보 연결성을 개선하는 방향이 적절하다.

## 12. 나라장터 레퍼런스 반영 포인트

실제 나라장터 입찰공고목록 화면을 기준으로 아래 패턴을 유지하는 것이 적절하다.

- 상단에 업무 맥락이 드러나는 제목과 브레드크럼을 둔다.
- 검색 영역은 테이블 위 고정 배치하고, 기본 조건과 상세 조건을 분리한다.
- 리스트는 카드형보다 고밀도 테이블이 적합하다.
- 공고번호, 공고명, 공고기관, 수요기관, 게시일시, 마감일시처럼 실무자가 반복 확인하는 컬럼을 중심으로 둔다.
- 엑셀 다운로드, 페이지 크기, 정렬 같은 테이블 도구막대는 우측 상단에 배치한다.
- 상태/단계/진행 컬럼은 배지나 짧은 태그 형태로 압축 표현한다.
- 원본 나라장터의 우측 고정 퀵메뉴는 축소 적용하거나, 내부 시스템 목적에 맞게 운영 액션 중심으로 재해석한다.
