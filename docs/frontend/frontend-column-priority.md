# Frontend Column Priority

## 1. 목적

- 본 문서는 화면별 테이블 컬럼 우선순위를 정의한다.
- 목표는 정보 과밀을 제어하면서도 실무자가 가장 자주 보는 값을 먼저 노출하는 것이다.

## 2. 우선순위 기준

### P1

- 항상 기본 노출
- 화면의 핵심 판단에 직접 필요

### P2

- 기본 노출 가능
- 화면 폭이 좁거나 밀도가 높으면 축약/토글 가능

### P3

- 기본 숨김 또는 확장영역/Drawer 이동
- 필요 시만 확인하는 보조 정보

## 3. 메인 입찰 공고 리스트

### 권장 컬럼 우선순위

| 컬럼 | 우선순위 | 이유 |
|---|---|---|
| 관심 여부 | P1 | 검토 대상 식별에 즉시 필요 |
| 상태 | P1 | 운영 상태를 가장 먼저 봄 |
| 업무구분 | P1 | 용역/물품/공사/외자 빠른 구분 |
| 공고번호 | P1 | 상세 조회와 내부 커뮤니케이션의 기준 키 |
| 공고명 | P1 | 실제 판단의 중심 |
| 공고기관 | P1 | 발주 성격 파악에 중요 |
| 수요기관 | P1 | 실제 사업 주체 확인에 중요 |
| 마감일시 | P1 | 실무 우선순위 판단 핵심 |
| 개찰일시 | P1 | 후속 추적 판단 핵심 |
| 추정가격 | P2 | 중요하지만 화면 폭을 많이 차지함 |
| 계약체결방법 | P2 | 자격/전략 판단에 유용 |
| 공고일시 | P2 | 최근성 판단용 |
| 변경 여부 | P2 | 변경 추적에 중요 |
| 첨부 존재 | P2 | 상세 검토 전 유용 |
| 재공고 여부 | P3 | 배지나 아이콘으로 축약 가능 |
| 통합공고번호 | P3 | 보조 연결 키 |
| 사전규격등록번호 | P3 | 보조 연결 키 |

### 기본 표시 권장 조합

- 관심 여부
- 상태
- 업무구분
- 공고번호
- 공고명
- 공고기관
- 수요기관
- 마감일시
- 개찰일시
- 추정가격

## 4. 사전 탐색 리스트

### 권장 컬럼 우선순위

| 컬럼 | 우선순위 | 이유 |
|---|---|---|
| 단계구분 | P1 | 발주계획/사전규격/조달요청 구분 핵심 |
| 업무구분 | P1 | 사업 성격 파악 |
| 사업명 | P1 | 검토 중심 컬럼 |
| 사업번호 | P1 | 추적 기준 |
| 사업일자 | P1 | 최근성 판단 |
| 공고/계약기관 | P2 | 기관 파악 |
| 수요기관 | P2 | 실제 수요 주체 확인 |
| 본공고 연결 여부 | P1 | 사전 탐색의 핵심 판단값 |
| 진행 상태 | P2 | 단계 진입 여부 확인 |
| 더보기 액션 | P1 | 확장행 진입 액션 |

### 설계 메모

- 사전 탐색은 컬럼 수보다 확장행 타임라인이 더 중요하다.
- 따라서 1차 테이블은 요약형으로 유지하고, 상세 연결은 확장행에서 해결한다.

| 요소 | 우선순위 | 이유 |
|---|---|---|
| 단계명 | P1 | 카드 식별 핵심 |

## 8. 모바일/좁은 화면 축약 순서

### 메인 리스트에서 먼저 숨길 컬럼

1. 계약체결방법
2. 공고일시
3. 첨부 존재
4. 변경 여부
5. 추정가격

### 끝까지 유지할 컬럼

1. 상태
2. 공고번호
3. 공고명
4. 기관명
5. 마감일시

## 9. 설계 결론

- 메인 리스트는 `공고명 + 기관 + 일정 + 상태` 중심으로 설계한다.
- 사전 탐색은 `요약 컬럼 + 확장행 타임라인` 조합이 적절하다.
- 사후 분석은 `결과값 중심`으로 재구성한다.
- 모든 화면에서 연결 키는 유지하되, 보조 키는 숨김/확장 영역으로 보내는 것이 가장 효율적이다.
