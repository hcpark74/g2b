# API Sync Strategy

## 1. 목적

- 본 문서는 G2B 외부 API의 우선순위, 배치 수집 순서, 호출 빈도, 재시도 정책을 정의한다.
- 기준 문서는 `docs/PRD.md`, `docs/architecture/system-architecture.md`, `docs/api/` 하위 명세를 따른다.

## 2. 기본 원칙

- 운영 중심 수집은 `입찰공고정보서비스`를 기준으로 한다.
- 타임라인 보강은 `계약과정통합공개서비스`로 수행한다.
- 결과 확정은 `낙찰정보서비스`, `계약정보서비스`로 보강한다.
- 사전 탐색 데이터는 본공고 전환 가능성이 높은 데이터만 선별 저장한다.
- API 트래픽 제한을 고려해 전체 동기화와 개별 재수집을 분리 운영한다.

### 입찰공고정보서비스 중심 원칙

- 내부 `bids` 마스터는 `BidPublicInfoService` 기준으로 생성한다.
- 다른 외부 API는 기본적으로 `bidNtceNo` 또는 그 보조 연결 키를 통해 이 마스터를 보강한다.
- 메인 리스트, 마감일, 공고 상태, 기관 정보, 1차 첨부 URL은 우선 이 서비스에서 확보한다.
- 본공고 이전 단계 데이터라도 `bfSpecRgstNo`, `orderPlanUntyNo`, `untyNtceNo`를 이 서비스에서 확보하면 내부 연결성이 크게 좋아진다.

## 3. API 우선순위

### Tier 1: 운영 필수

1. `입찰공고정보서비스`
2. `계약과정통합공개서비스`
3. `낙찰정보서비스`
4. `계약정보서비스`

특징:

- 메인 리스트, 상세 상태, 결과 확정, 타임라인 표시를 구성하는 핵심 계층
- MVP에서는 이 4개만으로도 운영형 대시보드의 핵심 기능 구현 가능
- 이 중에서도 `입찰공고정보서비스`는 선행 수집 필수 계층이다.

### 입찰공고정보서비스 내부 우선순위

#### 1차 수집

- 업무구분별 기본 목록 조회
  - `getBidPblancListInfoCnstwk`
  - `getBidPblancListInfoServc`
  - `getBidPblancListInfoFrgcpt`
  - `getBidPblancListInfoThng`

역할:

- 공고 마스터 생성
- 메인 리스트 구성
- 공고번호, 기관, 마감일, 공고명, 계약체결방법, 공고상태 확보

#### 2차 보강

- `getBidPblancListInfoLicenseLimit`
- `getBidPblancListInfoPrtcptPsblRgn`
- `getBidPblancListInfoEorderAtchFileInfo`
- `getBidPblancListInfoThngPurchsObjPrdct`
- `getBidPblancListInfoServcPurchsObjPrdct`
- `getBidPblancListInfoFrgcptPurchsObjPrdct`

역할:

- 자격/지역/세부품목/첨부파일 상세 보강

#### 3차 고도화

- `getBidPblancListInfo*PPSSrch`
- `getBidPblancListInfo*BsisAmount`
- `getBidPblancListInfoChgHstry*`
- `getBidPblancListBidPrceCalclAInfo`
- `getBidPblancListEvaluationIndstrytyMfrcInfo`

역할:

- 고급 검색, 가격 분석, 변경 추적, 공사 평가기준 분석

### Tier 2: 사전 탐색 및 연결 보강

5. `사전규격정보서비스`
6. `발주계획현황서비스`
7. `조달요청서비스`

특징:

- 공고 전 단계 리드 발굴과 연결 키 확보에 유용
- 본공고 중심 운영이 안정화된 뒤 비중을 높이는 것이 좋음

### Tier 3: 전략/분석 보조

8. `업종 및 근거법규서비스`
9. `가격정보현황서비스`
10. `사용자정보 서비스`

특징:

- 참여 자격, 가격 판단, 기관/업체 분석 등 보조 판단용
- 실시간성보다 기준정보/분석 정확도가 중요

## 4. 배치 수집 전략

### 4.1 일일 기본 배치

목표:

- 업무 시작 전 신규 공고와 상태 변경 공고를 반영한다.

권장 순서:

1. `입찰공고정보서비스`
2. `사전규격정보서비스`
3. `낙찰정보서비스`
4. `계약정보서비스`
5. `계약과정통합공개서비스`

이유:

- 먼저 메인 공고 목록을 확보해야 내부 대상 집합이 결정된다.
- 사전규격은 본공고 연결 가능성을 높인다.
- 낙찰/계약은 결과 확정용이다.
- 계약과정통합공개는 마지막에 타임라인 보강용으로 쓰는 것이 중복 호출을 줄이기 쉽다.

#### 입찰공고정보서비스 세부 순서

1. 기본 목록 4종 수집 (`공사/용역/외자/물품`)
2. 신규 공고/변경 공고 집합 생성
3. 필요 시 `기타공고` 포함 여부 판단
4. 신규 또는 관심 공고에 대해 면허제한/참가가능지역/구매대상물품 보강
5. 상세 첨부파일 및 e발주 첨부 보강

권장 규칙:

- 일괄 수집은 기본 목록 조회를 먼저 끝낸 뒤, 상세/보강 계열을 후속 배치로 분리한다.
- 모든 공고에 대해 25개 오퍼레이션을 전부 호출하지 않는다.
- 상세 오퍼레이션은 신규 공고, 변경 공고, 관심 공고에만 선별 적용한다.

### 4.2 관심 공고 보강 배치

목표:

- 즐겨찾기 또는 검토중 상태 공고를 더 자주 추적한다.

권장 대상:

- `status in ('reviewing', 'favorite', 'submitted')`
- 마감일 임박 공고
- 최근 7일 내 상태가 변경된 공고

권장 순서:

1. `입찰공고정보서비스` 상세/보강 계열
2. `계약과정통합공개서비스`
3. `낙찰정보서비스`
4. `계약정보서비스`

이유:

- 관심 공고는 상태 전이 확인이 중요하지만, 동시에 공고 자체 변경도 놓치면 안 된다.
- 따라서 변경이력/면허제한/첨부 보강이 가능한 `입찰공고정보서비스` 상세 계열을 먼저 고려한다.
- 이후 통합 추적과 결과/계약 보강을 수행한다.

### 4.3 기준정보 배치

목표:

- 실시간보다 일관성과 정합성이 중요한 기준정보를 주기적으로 갱신한다.

대상:

- `업종 및 근거법규서비스`
- 필요 시 `가격정보현황서비스`, `사용자정보 서비스`

권장 주기:

- 업종/근거법규: 일 1회 또는 주 1회
- 가격/사용자정보: 주기 재검토 후 별도 설정

## 5. 실시간/수동 재수집 전략

### 사용자 액션 기반 재수집

- 상세 Drawer에서 `다시 읽어오기` 실행 시 해당 공고 1건만 재수집
- 이때 전체 리스트 API보다 상태 확인용 API를 우선 호출

권장 순서:

1. `입찰공고정보서비스`
2. `계약과정통합공개서비스`
3. `낙찰정보서비스`
4. `계약정보서비스`

이유:

- 수동 재수집은 사용자가 보고 있는 공고의 최신 본문 상태, 첨부 URL, 마감 정보가 우선이다.
- 따라서 운영 마스터 소스인 `입찰공고정보서비스`를 먼저 확인한다.

### 자동 재수집 트리거

- 마감일 경과 직후
- 개찰 예정 시간 경과 후
- 낙찰/계약 미확정 상태 장기 지속 시

## 6. 계약과정통합공개서비스 호출 전략

### 역할

- 타임라인 연결 보강
- 단계 간 연결 키 재확인
- 낙찰/계약 서브데이터셋 확보

### 호출 우선순위

1. `inqryDiv=1` + `bidNtceNo`
2. `inqryDiv=2` + `bfSpecRgstNo`
3. `inqryDiv=3` + `orderPlanNo`
4. `inqryDiv=4` + `prcrmntReqNo`

### 권장 정책

- 기본적으로 `bidNtceNo`가 있으면 그 값으로 먼저 조회
- 실패 또는 연결 불충분 시 보조 키 순차 재시도
- 동일 공고에 대해 4개 키를 매번 전부 호출하지 않는다

## 7. 입찰공고정보서비스 호출 전략

### 기본 조회 키

- `inqryDiv=1`: 등록일시
- `inqryDiv=2`: 입찰공고번호
- `inqryDiv=3`: 변경일시

### 권장 사용 순서

1. 일일 배치: `inqryDiv=1` 등록일시 기준
2. 개별 상세 조회: `inqryDiv=2` 공고번호 기준
3. 변경 추적 보강: `inqryDiv=3` 변경일시 기준 또는 변경이력 API 병행

### 상세 보강 키 정책

- `LicenseLimit`, `PrtcptPsblRgn`, `EorderAtchFileInfo`, `*PurchsObjPrdct` 계열은 현재 `bidNtceNo` 기반 상세 조회로 본다.
- 구현상 fallback 은 notice 계열 후보로만 제한한다: `bid_no -> raw_api_data.bidNtceNo -> raw_api_data.untyNtceNo`.
- `bfSpecRgstNo`, `orderPlanNo`, `prcrmntReqNo` 는 입찰공고정보 상세 보강 요청 파라미터로 직접 사용하지 않는다.
- 이 키들은 계약과정통합공개, 사전 단계 연결, 후속 추적용 보조 연결 키로 관리한다.

### 업무구분별 수집 순서

1. `용역`
2. `물품`
3. `공사`
4. `외자`
5. `기타`

권장 이유:

- Woorisoft의 주력 사업 영역이 AI, 소프트웨어, 자동화, 클라우드에 가까우므로 `용역`, `물품` 우선이 실무 적합성이 높다.
- `공사`, `외자`는 중요하지만 상대적으로 후순위로 배치 가능하다.
- `기타`는 데이터 성격이 넓고 불규칙성이 있으므로 초기에는 선택적으로 포함하는 것이 좋다.

## 8. 트래픽 관리 전략

### 기본 원칙

- 일일 트래픽이 `1000`인 API는 전체 스캔형으로 과도하게 호출하지 않는다.
- 배치 대상 집합을 먼저 좁힌 뒤 상세/보강 API를 호출한다.
- 중복 재조회 방지를 위해 최근 동기화 시각을 저장한다.

### 권장 제어 방식

- 공고별 `last_synced_at` 저장
- API별 `last_success_at`, `last_error_at` 저장
- 같은 공고에 대한 짧은 시간 내 반복 호출 제한
- 동일 키로 최근 성공한 응답은 캐시 우선 사용

### 입찰공고정보서비스에 대한 추가 원칙

- 목록 계열과 상세 계열의 호출 예산을 분리 관리한다.
- 검색조건 계열은 사용자 실시간 입력마다 직접 호출하지 않는다.
- 변경이력 계열은 전체 공고 대상이 아니라 변경 의심 공고에만 선별 적용한다.

### 동시성 정책

- SQLite 기반 초기 운영: 낮은 동시성
- Playwright와 API 호출은 같은 시각에 과도하게 몰지 않음
- 외부 API는 서비스별 rate limit 버킷으로 제어하는 것이 바람직함

## 9. 실패 처리 및 재시도

### 재시도 대상

- 네트워크 오류
- `HTTP Error`
- `service time out`
- 일시적 `Application Error`, `DB Error`

### 즉시 재시도하지 말아야 할 경우

- `03 No Data`
- `20 서비스 접근 거부`
- `22 요청 횟수 초과`
- `30 등록되지 않은 서비스 키`
- `31 기한 만료된 서비스 키`
- `32 등록되지 않은 도메인/IP`

### 권장 재시도 정책

- 1차 실패: 짧은 지수 백오프
- 2차 실패: 동일 배치 내 후순위로 미룸
- 반복 실패: `sync_jobs` 또는 별도 에러 로그에 기록 후 운영 확인 대상으로 분리

입찰공고정보서비스 특화 메모:

- 기본 목록 조회 실패는 전체 파이프라인 영향도가 크므로 우선 재시도한다.
- 상세 보강 계열 실패는 부분 실패로 분리하고 메인 배치를 막지 않는다.

## 10. 단계별 저장 전략

### 메인 적재

- `입찰공고정보서비스` -> `bids`, `bid_details`

세부 적재:

- 기본 목록 계열 -> `bids`
- 첨부 URL/파일명 -> `bid_details` 또는 별도 첨부 메타 레이어
- 면허제한/참가가능지역 -> 별도 정규화 테이블 후보
- 구매대상물품 -> 품목 정규화 테이블 후보
- 변경이력 -> `timeline_logs` 또는 별도 변경이력 테이블 후보

### 보강 적재

- `계약과정통합공개서비스` -> `contract_process_integrations`
- `bidwinrInfoList` -> `contract_process_awards`
- `cntrctInfoList` -> `contract_process_contracts`

### 상태 확정 적재

- `낙찰정보서비스` -> 결과 분석 테이블 또는 낙찰 상태 필드
- `계약정보서비스` -> 계약 상태 필드 및 계약 상세 레이어

### 기준정보 적재

- `업종 및 근거법규서비스` -> `industry_base_law_rules`, `industry_inclusion_licenses`

## 11. 권장 운영 시나리오

### 시나리오 A: 아침 일괄 수집

- 신규/변경 공고 수집
- 결과/계약 상태 보강
- 타임라인 보강

세부 흐름:

1. `입찰공고정보서비스` 기본 목록 수집
2. 신규/변경 공고 식별
3. 필요한 공고만 상세 보강
4. 결과/계약 보강
5. 타임라인 보강

### 시나리오 B: 실무자 상세 검토

- 특정 공고 열람
- 수동 재수집 실행
- 타임라인과 낙찰/계약 상태 즉시 갱신

세부 흐름:

1. `입찰공고정보서비스`로 최신 공고 상태 재확인
2. 면허/지역/첨부/품목 정보 보강
3. 통합 타임라인 조회
4. 낙찰/계약 정보 보강

### 시나리오 C: 전략 분석

- 낙찰률 집계
- 계약 체결 패턴 분석
- 업종 자격 검토 및 대응 전략 수립

## 12. 결론

- 초기 운영은 `입찰공고정보서비스 + 계약과정통합공개서비스 + 낙찰정보서비스 + 계약정보서비스` 중심으로 설계하는 것이 가장 효율적이다.
- 사전 탐색 API는 리드 파이프라인 강화용으로 2차 비중을 높인다.
- 기준정보 API는 실시간 처리보다 배치성 동기화가 적합하다.
- 통합 추적 API는 모든 화면의 원천이 아니라, 핵심 운영 데이터를 연결하고 보강하는 계층으로 쓰는 것이 바람직하다.
- 특히 `입찰공고정보서비스`는 전체 수집 파이프라인의 출발점이자 내부 `bids` 마스터의 기준 원천으로 고정하는 것이 적절하다.
