# Frontend Stage Card Spec

## 1. 목적

- 본 문서는 `타임라인 탭`과 `행 확장형 타임라인`에 공통으로 사용하는 단계 카드 UI 명세를 정의한다.
- 목표는 사전규격부터 계약까지의 단계를 한눈에 비교 가능하게 만드는 것이다.

## 2. 적용 위치

- 공고 상세 Drawer의 `타임라인` 탭
- 사전 탐색/통합 검색의 `확장행 타임라인`
- 필요 시 대시보드 요약 카드의 축약형 단계 표시

## 3. 공통 구조

각 단계 카드는 동일한 기본 구조를 가진다.

```text
+--------------------------------+
| 단계명                상태배지 |
|--------------------------------|
| 대표 번호                        |
| 대표 일시                        |
| 보조 정보 1                      |
| 보조 정보 2                      |
| [상세보기]                       |
+--------------------------------+
```

공통 요소:

- 단계명
- 상태 배지
- 대표 식별번호
- 대표 일시
- 보조 메타 정보
- 선택적 액션 또는 링크

## 4. 단계 정의

권장 단계 순서:

1. `발주계획`
2. `요청접수`
3. `사전규격`
4. `입찰공고`
5. `개찰/낙찰`
6. `계약`
7. `검사검수` (선택)
8. `대금지급` (선택)

설계 메모:

- MVP에서는 `사전규격`, `입찰공고`, `개찰/낙찰`, `계약` 4단계를 우선 구현한다.
- `발주계획`, `요청접수`는 선행 데이터가 확보되면 앞단에 붙인다.
- `검사검수`, `대금지급`은 내부 운영 범위와 데이터 가용성에 따라 후순위로 둔다.

## 5. 단계별 카드 명세

### 5.1 발주계획 카드

표시 필드:

- 단계명: `발주계획`
- 대표 번호: `orderPlanNo` 또는 `orderPlanUntyNo`
- 대표 일시: 발주년월 또는 발주일자
- 보조 정보:
  - 발주사업명
  - 발주기관명

상태 규칙:

- 값 존재: `완료`
- 값 없음: `미도달`

### 5.2 요청접수 카드

표시 필드:

- 단계명: `요청접수`
- 대표 번호: `prcrmntReqNo`
- 대표 일시: 요청 접수일시
- 보조 정보:
  - 수요기관명
  - 요청 상태 또는 요약 문구

상태 규칙:

- 요청번호 존재: `완료`
- 번호 없음: `미도달`

### 5.3 사전규격 카드

표시 필드:

- 단계명: `사전규격`
- 대표 번호: `bfSpecRgstNo`
- 대표 일시: 사전규격 등록일시 또는 의견등록마감일시
- 보조 정보:
  - 사전규격사업명
  - 사전규격 공고기관명/수요기관명

상태 규칙:

- 사전규격번호 존재: `완료`
- 번호 없음: `미도달`

### 5.4 입찰공고 카드

표시 필드:

- 단계명: `입찰공고`
- 대표 번호: `bidNtceNo` + `bidNtceOrd`
- 대표 일시: `bidNtceDt`
- 보조 정보:
  - 공고명
  - 입찰마감일시
  - 개찰예정일시

상태 규칙:

- 공고 존재 + 마감 전: `진행중`
- 공고 존재 + 마감 후: `마감`
- 공고 없음: `미도달`

### 5.5 개찰/낙찰 카드

표시 필드:

- 단계명: `개찰/낙찰`
- 대표 번호: 낙찰 업체 사업자번호 또는 낙찰 데이터 순번
- 대표 일시: 개찰일시 또는 낙찰 공개일시
- 보조 정보:
  - 낙찰업체명
  - 낙찰금액
  - 낙찰률
  - 참가업체수

데이터 출처:

- `낙찰정보서비스`
- `계약과정통합공개서비스.bidwinrInfoList`

상태 규칙:

- 낙찰자 정보 존재: `완료`
- 개찰일은 지났지만 낙찰 정보 없음: `확인 필요`
- 개찰 전: `미도달`

### 5.6 계약 카드

표시 필드:

- 단계명: `계약`
- 대표 번호: 계약번호
- 대표 일시: 계약일자
- 보조 정보:
  - 계약명
  - 계약기관명
  - 계약금액
  - 계약체결방법명

데이터 출처:

- `계약정보서비스`
- `계약과정통합공개서비스.cntrctInfoList`

상태 규칙:

- 계약 정보 존재: `완료`
- 낙찰 후 계약 없음: `미체결`
- 낙찰 전: `미도달`

### 5.7 검사검수 카드

표시 필드:

- 단계명: `검사검수`
- 대표 번호: 검사 접수번호 또는 통합 접수번호
- 대표 일시: 검사일자
- 보조 정보:
  - 검사 상태
  - 검사 기관

상태 규칙:

- 데이터 존재: `완료`
- 없으면: `미도달`

### 5.8 대금지급 카드

표시 필드:

- 단계명: `대금지급`
- 대표 번호: 지급번호 또는 통합지급번호
- 대표 일시: 지급요청일시 또는 지급일
- 보조 정보:
  - 지급 상태
  - 지급 금액

상태 규칙:

- 데이터 존재: `완료`
- 없으면: `미도달`

## 6. 상태 배지 규칙

권장 상태:

- `완료`
- `진행중`
- `미도달`
- `미체결`
- `확인 필요`

표현 원칙:

- `완료`: muted green
- `진행중`: blue
- `미도달`: gray
- `미체결`: amber
- `확인 필요`: orange-red

## 7. 빈값 표현 규칙

표시 방식:

- 번호 없음 -> `없음`
- 일시 없음 -> `없음`
- 단계 자체 미도달 -> 카드 유지 + 회색 상태

설계 메모:

- 카드를 숨기면 흐름 이해가 어려워지므로, 단계는 유지하고 값만 비워 두는 방식이 좋다.

## 8. Drawer용 표현 규칙

- 정보 밀도를 높이되 2열 키-값 형태를 기본으로 사용
- 필요 시 각 단계 카드 아래에 이벤트 리스트를 추가
- 상세 링크는 `원문 보기`, `관련 첨부 보기`처럼 명시적 라벨 사용

## 9. 확장행용 표현 규칙

- 카드 수가 많으므로 폭을 줄이고 핵심 2~3줄만 노출
- 대표 번호와 대표 일시를 우선 노출
- 보조 정보는 1줄 요약으로 제한
- `더보기` 또는 `상세` 클릭 시 Drawer로 넘길 수 있음

## 10. 데이터 바인딩 우선순위

### 1차 바인딩

- `bid_id`
- `bidNtceNo`
- `bidNtceOrd`
- `bfSpecRgstNo`
- `orderPlanNo`
- `prcrmntReqNo`

### 2차 바인딩

- 낙찰/계약 정규화 테이블
- 타임라인 스냅샷 테이블

설계 메모:

- 카드 렌더링은 원본 API 응답보다 정규화된 내부 데이터 기준으로 하는 것이 안정적이다.

## 11. 구현 우선순위

### 1차

- 사전규격 카드
- 입찰공고 카드
- 개찰/낙찰 카드
- 계약 카드

### 2차

- 발주계획 카드
- 요청접수 카드

### 3차

- 검사검수 카드
- 대금지급 카드

## 12. 결론

- 단계 카드는 `번호 + 일시 + 상태` 3요소를 공통 축으로 유지하는 것이 가장 중요하다.
- 메인 운영은 Drawer 중심, 비교형 화면은 확장행 중심으로 쓰되 같은 카드 명세를 공유해야 일관성이 유지된다.
